# Site and wiki boundaries

The project uses two documentation tracks.

The MkDocs site is the canonical public documentation surface. The GitHub wiki is project memory.

Both are useful, but they have different jobs.

## MkDocs site

The MkDocs site should contain curated documentation that a contributor, reviewer or article reader can rely on as the current public project position.

Use the site for:

- the project thesis;
- the manual IssueOps workflow;
- execution-contract guidance;
- pull request evidence-pack guidance;
- contract verification guidance;
- safe tool-operation guidance;
- manual label guidance;
- stable examples;
- release notes; and
- the public roadmap.

Content in the site should be accurate, concise and maintained with the repository.

## GitHub wiki

The wiki should remain useful for broader project memory. It can hold material that is valuable but not yet canonical.

Use the wiki for:

- working notes;
- draft ideas;
- broader research;
- implementation reports;
- meeting notes;
- exploratory governance thinking; and
- historical context that is not part of the curated site.

Wiki content may inform future documentation work, but it should not be treated as canonical unless it has been moved or copied into the MkDocs site through an issue and pull request.

## Boundary rule

When in doubt, ask whether the reader should treat the content as the current public project position.

If yes, it belongs in MkDocs.

If no, it belongs in the wiki or another project-memory location until it is ready to be promoted.
